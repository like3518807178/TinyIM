## 今日完成

### 1. 代码结构重构
把原先更偏脚本式的服务端逻辑，拆成了两个更清晰的模块：
- `TcpServer`：负责创建 socket、绑定端口、监听、接收连接、收发消息
- `Logger`：负责统一打印日志，并带上时间戳

对应文件：
- `include/TcpServer.h`
- `src/TcpServer.cpp`
- `include/Logger.h`
- `src/Logger.cpp`

### 2. 服务端改为循环 `accept`
当前服务端运行逻辑分成两层循环：
- 外层循环：持续调用 `accept()`，等待新的客户端连接
- 内层循环：针对当前客户端反复 `recv()` / `send()`，直到客户端断开

这样做的结果是：
- 某个客户端断开，只会结束当前连接处理
- 服务端进程不会退出
- 处理完一个连接后，还会回到 `accept()`，继续等待下一个客户端

### 3. 日志增加时间戳
`Logger` 中通过时间格式化输出日志，日志形态类似：

```text
[2026-03-20 19:30:20] [INFO] Server started on port 9999
```

这样后面排查问题时，可以更清楚地看到：
- 服务端什么时候启动
- 客户端什么时候连接/断开
- 某次错误大概发生在什么时间点

## 验收结果
本日目标对应的核心验收已经满足：
- 客户端断开后，服务端不会退出
- 服务端可以继续接受后续连接
- 连续多次重连后仍然可以正常收发

## 网络基础笔记

### 1. 监听 fd 是什么
`listen_fd` 是监听 socket 对应的文件描述符。

它的职责不是直接和某个具体客户端通信，而是：
- 绑定服务端端口
- 进入监听状态
- 等待新的客户端连接到来

可以把它理解成“服务端的大门口”。

### 2. 连接 fd 是什么
`conn_fd` 是 `accept()` 成功之后返回的一个新的文件描述符。

这里一定要注意：
- `accept()` 返回的是“新 fd”
- 这个新 fd 才是和某个具体客户端通信用的 fd
- 后续的 `recv()`、`send()` 都应该针对这个 `conn_fd`，而不是 `listen_fd`

也就是说：
- `listen_fd` 负责等连接
- `conn_fd` 负责收发数据

### 3. `accept()` 的行为
`accept(listen_fd, ...)` 的行为可以概括成两点：
- 如果暂时没有新的客户端连接，它会阻塞等待
- 一旦有新的客户端连进来，它会返回一个新的连接 fd

这个“新 fd”非常关键，因为它代表一条已经建立好的 TCP 连接。

所以典型流程是：
1. `socket()` 创建监听 socket
2. `bind()` 绑定端口
3. `listen()` 开始监听
4. `accept()` 接受连接，拿到新的 `conn_fd`
5. 用 `conn_fd` 调 `recv()` / `send()`
6. 通信结束后关闭 `conn_fd`
7. 回到 `accept()` 继续等下一个客户端

### 4. 为什么客户端断开后服务端不该退出
因为客户端断开，影响的是当前连接，也就是 `conn_fd`。

正常做法是：
- 当前连接结束后，关闭这个 `conn_fd`
- 但保留 `listen_fd`
- 服务端继续在监听端口上等待新的连接

如果因为一个客户端断开就整个服务端退出，说明监听和连接这两个层次没有分清。

## 今天的收获
- 第一次把最小可跑的 server 抽成了类
- 对 `listen_fd` 和 `conn_fd` 的职责区分更清楚了
- 明白了 `accept()` 返回的是一个“新的连接 fd”，而不是把监听 fd 变成已连接 fd

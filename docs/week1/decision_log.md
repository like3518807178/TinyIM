# Decision Log

## 2026-03-18: 服务端先拆成 `TcpServer` 和 `Logger`

### 决策
先把最小 server 的启动、网络逻辑、日志输出拆开，而不是继续把代码堆在 `main.cpp`。

### 原因
- 后续要继续加配置、协议、连接状态，如果不先拆模块，代码会很快失控
- 日志属于横切能力，单独抽出来更容易复用

### 结果
- `main.cpp` 负责组装流程
- `TcpServer` 负责 socket 生命周期
- `Logger` 负责统一日志输出

## 2026-03-19: 配置从硬编码改为 `conf/server.conf`

### 决策
端口和日志级别不再写死在代码里，通过 `Config` 模块读取配置文件。

### 原因
- 改端口不应该每次都重新改源码
- 为后续继续增加配置项留出入口

### 结果
- 新增 `Config` 类
- `main.cpp` 启动时先加载配置，再初始化日志和服务端

## 2026-03-20: 应用层协议采用 4 字节长度前缀

### 决策
消息格式使用“4 字节长度 + payload”，而不是把每次 `recv()` 直接当成一条完整消息。

### 原因
- TCP 是字节流，不自带消息边界
- 不加协议就无法正确处理半包和粘包

### 结果
- `TcpServer` 里增加拆帧逻辑
- 每个连接需要维护输入缓冲区

## 2026-03-21: socket 全部切成非阻塞，并补充输出缓冲区

### 决策
监听 socket 和连接 socket 一起改为非阻塞，读写都按“多轮完成”的模型处理。

### 原因
- 阻塞式 `accept/recv/send` 会把整个主循环卡住
- 后续如果要上 `epoll`，必须先把非阻塞 I/O 语义理顺

### 结果
- 新增 `SetNonBlocking`
- 连接状态中增加 `input_buffer` 和 `output_buffer`
- `HandleRead()` / `HandleWrite()` 负责处理 `EINTR`、`EAGAIN` 和部分发送

## 2026-03-22: 仓库统一使用 `build/`，不再提交构建产物

### 决策
删除仓库中的 `build/`、`buid/` 生成文件，新增 `.gitignore`，对外文档统一使用 `build/`。

### 原因
- 构建产物进仓库会干扰阅读
- `buid` 是命名错误，继续保留只会让协作者困惑
- README 里必须让“怎么编译/怎么跑”足够直接

### 结果
- 目录命名收敛为 `build/`
- README 按首次阅读者视角重写
- 启动逻辑兼容根目录和 `build/` 目录两种运行方式

## 脚本怎么工作

### 协议格式
客户端发送和接收都遵循当前 TinyIM 协议：
- 4 字节 `cmd`
- 4 字节 `request_id`
- 4 字节 `body_len`
- `body` 字节流

### 并发模型
脚本的思路是：
1. 起多个线程
2. 每个线程建立一个 TCP 连接
3. 每个连接发送多条请求
4. 根据 `--burst` 控制“先发几条，再逐步收响应”
5. 收到响应后校验是否和对应请求匹配

### 为什么 `--burst` 有意义
如果只是“一发一收”，虽然能测多连接，但对单连接内的缓冲、协议解析、写回路径压力还不够大。

加入 `--burst` 后：
- 一个连接里可以连续发多条包
- 服务端需要连续读取、解析多帧
- 响应也需要在多请求场景下正确对齐

这更接近真实 IM 连接的行为。

## 使用方式

### 1. 先编译并启动服务端
```bash
cmake -S . -B build
cmake --build build
./build/TinyIM
```

### 2. 另开一个终端运行并发测试
```bash
python3 scripts/verify_concurrent_protocol.py --host 127.0.0.1 --port 9999 --clients 20 --requests 20 --burst 5
```

### 3. 结果怎么看
如果成功，会看到每个连接的结果类似：
```text
(0, True, '20 requests matched')
```

最后还会看到汇总：
```text
SUMMARY: passed=20/20, requests_per_client=20, burst=5
```

这表示：
- 一共 20 个并发连接
- 每个连接发了 20 条请求
- 每个连接最多允许 5 条请求在途
- 所有连接的响应都校验通过

## 参数说明
- `--host`：服务端地址，默认 `127.0.0.1`
- `--port`：服务端端口，默认 `9999`
- `--clients`：并发连接数
- `--requests`：每个连接发送多少条请求
- `--burst`：每个连接允许多少条“已发送但还没收到响应”的请求

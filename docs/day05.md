# Day 05

日期：2026-03-21

## 今日目标
- 把监听 socket 和连接 socket 全部改为非阻塞模式
- 读操作改成“持续读到 `EAGAIN/EWOULDBLOCK` 为止”
- 写操作改成“尽量发送，写不动时保留剩余数据，下轮继续”
- 补齐错误处理和日志，避免高频发送时崩溃或死循环

## 今天为什么必须做非阻塞
前一天虽然已经有了长度前缀协议和输入缓冲区，但当前服务端还是阻塞式的：
- `accept()` 没有新连接时会一直卡住
- `recv()` 遇到客户端慢发时会一直卡住
- `send()` 遇到内核发送缓冲区满时也可能一直卡住

这意味着只要某个阶段堵住，整个服务端主循环就被拖住了。

今天的目标不是一步上 `epoll`，而是先把 socket 行为改正确：
- 监听 fd 不阻塞
- 已连接 fd 不阻塞
- `EAGAIN/EWOULDBLOCK` 被当作“现在先没有更多可读/可写了”，而不是错误

这样后面切到 `epoll` 时，连接管理模型就不会推倒重来。

## 代码上的核心变化

### 1. 监听 fd 和连接 fd 全部设置 `O_NONBLOCK`
新增了 `SetNonBlocking(int fd)`：
- `Start()` 中对 `listen_fd_` 设置非阻塞
- `accept()` 成功拿到 `conn_fd` 后，立即对这个新连接 fd 设置非阻塞

这里再次强调一遍：
- `listen_fd_` 只负责接新连接
- `accept()` 返回的 `conn_fd` 才负责和某个客户端收发数据
- `accept()` 不会“修改监听 fd”，而是返回一个新的已连接 fd

### 2. 引入连接状态表
之前的代码是：
- 接一个连接
- 进入这个连接的内层 `recv/send` 死循环
- 处理完再回去 `accept()`

现在改成：
- 用 `connections_` 保存当前所有连接
- 每个连接保存三块状态：
  - `peer`
  - `input_buffer`
  - `output_buffer`

这样做的原因是：非阻塞下，一个连接的读写经常分多轮完成，不能再靠局部变量一次性处理完。

### 3. 读路径改成“读到 `EAGAIN`”
新增 `HandleRead()`，逻辑是：
1. `recv()` 成功读到数据，就追加进 `input_buffer`
2. 反复调用拆帧逻辑，尽量从缓冲区里取出完整消息
3. 如果 `recv()` 返回 `-1` 且 `errno == EAGAIN/EWOULDBLOCK`，说明这一轮已经读完了，不是错误
4. 如果 `recv()` 返回 `0`，说明对端关闭连接
5. 如果是其他错误，记录日志并关闭连接

这一步解决的问题是：
- 不会因为一次没数据可读就误判失败
- 不会在高频发送时只读一小段就退出本轮

### 4. 写路径改成“写到写不动”
之前 `SendFrame()` 的语义是“立刻把整帧全部发出去”，这是阻塞式写法。

现在改成两步：
- `SendFrame()` 只负责把打包后的帧追加到 `output_buffer`
- `HandleWrite()` 负责循环 `send()`

如果出现：
- `send()` 成功发送一部分，就从 `output_buffer` 头部擦掉已发送部分
- `send()` 返回 `-1` 且 `errno == EAGAIN/EWOULDBLOCK`，说明当前写不动了，保留剩余数据，下轮继续

这一步的关键意义是：
- 不会因为发送缓冲区一时写满就把连接当错误关闭
- 可以正确处理“部分发送”

### 5. 错误处理更细了
今天把错误处理从“统一打印 failed”补成了更接近真实排障的形式：
- 日志里带 `strerror(errno)`，知道具体失败原因
- `EINTR` 会重试
- `EAGAIN/EWOULDBLOCK` 会走正常分支
- 非法帧（比如 body 长度超过上限）会记录错误并关闭连接

## 当前主循环长什么样
主循环现在可以概括成三步：

1. 持续 `accept()`，直到监听 fd 返回 `EAGAIN`
2. 遍历当前连接表，对每个连接执行：
   - `HandleRead()`
   - `HandleWrite()`
3. 轻微休眠，避免空转占满 CPU

这本质上是一个“手写轮询版”的非阻塞服务端。

它的定位是：
- 先把非阻塞收发语义练清楚
- 为下一步 `select/poll/epoll` 打地基

## 验收标准
今天对应的验收点是：
- 高频发送下服务端不崩溃
- 服务端不会因为 `EAGAIN` 进入错误分支
- 服务端不会在无连接时死循环打满 CPU
- 客户端断开后，连接能被正确清理

## 我今天实际学到的点
- 非阻塞不等于“更快”，先意味着“你不能再假设一次系统调用就完成任务”
- `EAGAIN/EWOULDBLOCK` 在非阻塞模型里是正常状态，不是异常
- 写路径比读路径更容易忽略“部分发送”问题
- 没有 I/O 多路复用时，也可以先把连接状态机搭起来

## 还存在的局限
今天的版本已经比阻塞式稳很多，但它仍然不是最终形态：
- 现在还是手动轮询 `connections_`
- 为了避免空转，主循环里用了 `usleep(1000)`
- 连接数一多，这种方式效率不高

所以后续自然的下一步就是：
- 引入 `epoll`
- 让内核告诉我们“谁可读、谁可写”
- 不再靠用户态遍历所有连接

## 小结
Day 05 不是功能堆叠日，而是一次很关键的“网络语义纠偏”：
- 让监听 socket 和连接 socket 都进入非阻塞
- 让 `recv/send` 的返回值处理更符合真实服务器写法
- 让连接状态不再依赖一次调用完成，而是能跨多轮保存

这一步做完之后，TinyIM 才真正开始从“教学 demo”往“可继续演进的小型服务端”靠近。
